Reverse-engineering and runtime instrumentation

A case study in building tooling on top of an undocumented Unreal Engine 5 binary.

Lua · Python · PowerShell · about 13,300 lines

This was a personal reverse-engineering project. The goal was to observe and instrument a shipping UE5 game's runtime with no source code, no documentation, and no type information, and then build something stable enough to hand to non-technical users.

This write-up covers the engineering and the approach. The code itself is a personal research project that I keep private, but I'm happy to walk through it on request.

The constraints

The target is an opaque, shipping game binary. The engine internals aren't documented, the scripting layer it exposes has no type information, and the hard part is that native memory faults in the engine skip the scripting language's error handling completely. A bad call doesn't throw an error you can catch. It takes the whole process down and leaves nothing in the log to tell you why.

Catching crashes the runtime hides

Because native faults skip the scripting runtime's protected calls, a crash usually leaves no trace, and the host tooling even wipes its own log on restart. So the core of this project is a logging system that writes a running trail of breadcrumbs to disk and survives across sessions. When the game goes down, I can open that trail and see exactly where it died, after the fact.

On top of that diagnostic backbone sits a layer of defensive code. Each piece traces back to a real crash it was built to stop:

  • A protected wrapper around even the engine's own validity checks, so a stale handle can't fault the process.
  • A cached resolver for engine state that avoids unsafe calls during the dangerous map-transition window.
  • Clearing every stale object reference on each map load. Stale references are an instant crash, so they get cleared the moment the world changes.

The commit history shows the pattern over and over: a feature goes in, it crashes in the field, I trace it with the breadcrumb log, and then I defend against it.

Keeping the threads honest

External tooling talks to the in-game scripting layer over a file-based bridge. Touching engine objects off the game thread is undefined behavior, so the bridge loop stays separate from the game thread and object reads are cached and marshaled across. I learned that rule from an actual crash, not from a manual.

One system, four runtimes

The whole thing is one cross-language stack: embedded Lua running inside the game, a file-based bridge, an external Python overlay for the GUI, and a PowerShell layer for installation, builds, and diagnostics. Each piece does the job it's best at, and they're wired together into a single system.

Distribution that survives real users

A PowerShell installer finds Steam library locations on its own (it reads the Windows registry and parses libraryfolders.vdf), installs the scripting layer, verifies each step, and writes a diagnostic bundle if anything fails. A non-technical user can install it without help from me, and when something does break, it comes back as something I can actually debug.

What it shows

  • Reverse engineering: working productively in a system with no docs, no types, and no source.
  • Defensive systems programming: debugging and hardening software under crash-prone conditions.
  • Cross-language range: embedded scripting, IPC, a GUI, and packaging, all integrated and shipping.
