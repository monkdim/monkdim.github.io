Reverse-engineering & runtime instrumentation

A case study: building resilient tooling on top of an undocumented Unreal Engine 5 binary.

Lua · Python · PowerShell · ~13,300 lines

A personal reverse-engineering research project: instrument and observe a shipping UE5 game's runtime with no source, no documentation, and no type information — then build software resilient enough to survive in that hostile environment and ship to non-technical users.

This write-up covers the engineering and methodology. The code itself is a personal research project and is kept private; it's available to discuss on request.

The constraints

The target is an opaque, shipping game binary. The engine's internals are undocumented, the scripting layer it exposes has no type information, and — the hard part — native memory faults in the engine bypass the scripting language's error handling entirely. A bad call doesn't throw; it takes the whole process down, and leaves nothing in the log to explain why.

Crash-survival engineering

Because native faults skip the scripting runtime's protected-call mechanism, a crash normally leaves no trace — and the host tooling even wipes its own log on restart. So the centerpiece of this project is a persistent breadcrumb + heartbeat forensics system: a running trail written to disk that survives across sessions, letting me reconstruct exactly where a native crash happened, after the fact.

On top of that diagnostic backbone sits a layer of defensive engineering, each piece traceable to a real crash it was built to prevent:

  • A protected validity wrapper around even the engine's own validity checks, so a stale handle can't fault the process.
  • A cached resolver for engine state that avoids unsafe calls during the dangerous map-transition teardown window.
  • Comprehensive invalidation of stale object references on every map load — stale references are an instant crash, so they're cleared the moment the world changes.

The result reads, in the commit history, like software that was shipped, broke in the field, and got hardened: features tried, crashed, diagnosed via the breadcrumb trail, and then defended.

Concurrency discipline

External tooling communicates with the in-game scripting layer over a file-based IPC bridge. Touching engine objects off the game thread is undefined behavior, so the IPC loop is strictly separated from the game thread and object reads are cached and marshaled — a rule learned from an actual crash, not from theory.

One system, four runtimes

The whole thing is a coherent cross-language stack: embedded Lua running inside the game, a file-based IPC bridge, an external Python GUI overlay, and a PowerShell layer for installation, builds, and diagnostics. Each piece is doing the job it's best at, wired together end to end.

Distribution that survives the real world

A PowerShell installer auto-detects Steam library locations (Windows registry plus libraryfolders.vdf parsing), installs the scripting layer, verifies each step, and emits a diagnostic support bundle on failure — so a non-technical user can install it without hand-holding, and any failure comes back as something I can actually debug.

What it demonstrates

  • Reverse engineering — working productively in a system with no docs, no types, and no source.
  • Defensive systems programming — debugging and hardening software under adversarial, crash-prone runtime conditions.
  • Cross-language full-stack reach — embedded scripting ↔ IPC ↔ GUI ↔ packaging and deployment, all integrated and shipping.
